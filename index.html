WK 01 How the Internet Actually Works — APIs, JSON, HTTP ✅ 0/7 5–6 hrs ▼
Goal: Before touching any tools, understand the language everything speaks. APIs, HTTP requests, and JSON are the foundation of everything you'll build. This week is concept-first, then hands-on proof.
Concepts to Understand
  • What an API is: A set of rules that lets two software systems talk to each other. When Make.com sends data to Airtable, it uses Airtable's API. Think of it as a restaurant menu — you order from it, the kitchen responds. Read: "What is an API?" on HowStuffWorks (15 min)
  • HTTP requests: GET (fetch data), POST (send data), PUT (update), DELETE (remove). You'll mostly use GET and POST. Read the first 2 sections of MDN's "HTTP overview" — stop before "HTTP/2" (20 min)
  • JSON: The data format APIs speak. It's just key-value pairs: {"name": "Gavin", "service": "HVAC"}. Make.com passes JSON between modules. Write 3 sample JSON objects by hand — a job record, a customer record, an SMS message
  • Authentication: How APIs know who's calling them. API keys (a secret password in the request header) are what you'll use most. Bearer tokens and Basic auth are variations. Read Twilio's "What is an API key?" page (10 min)
Hands-On Proof
  • Install Hoppscotch (free, browser-based) at hoppscotch.io — this is an API testing tool. Make a GET request to: https://jsonplaceholder.typicode.com/posts/1 — you'll see a real JSON response come back. This is exactly what Make.com does behind the scenes.
  • Make a POST request in Hoppscotch to https://jsonplaceholder.typicode.com/posts with a JSON body: {"title": "Test Job", "body": "HVAC repair needed", "userId": 1}. See the response. You just made your first API call.
  • Write in your Journal: "An API is..." in your own words. Then: "JSON is..." Then: "When Make.com sends an SMS, what's actually happening under the hood?" Answer it yourself before moving to Week 2.

WK 02 Make.com Deep Dive — Triggers, Modules, and Data Flow ✅ 0/8 6–7 hrs ▼
Goal: Understand Make.com from the inside out — not just how to click buttons, but why it works the way it does. By end of week you can build any basic scenario from scratch and explain every decision.
How Make.com Actually Works
  • Triggers vs. Actions: Every scenario starts with one trigger (the event that fires it) and can have unlimited action modules after it. The trigger is always the first bubble. Create a free Make.com account if you haven't. Go to "Create a new scenario" and just look at the empty canvas for 2 minutes — notice the + button and the trigger placeholder.
  • Build Scenario #1 — Webhook to email: Add a "Webhooks" trigger (custom webhook). Copy the webhook URL Make gives you. Paste it into your browser address bar and add ?name=Gavin&service=HVAC at the end. Hit enter. Watch Make receive the data. Add a "Gmail" or "Email" action to send yourself the data. You just built a real trigger-action scenario.
  • Data mapping: In Make, when you click a field in one module, you can insert data from a previous module using the blue tokens. This is called mapping. In your email action, map the "name" and "service" values from the webhook into the email subject. This is the core skill in all of Make.
  • Filters and routers: Add a Filter between your webhook trigger and email action. Set it to only proceed if "service" equals "HVAC". Test it with service=HVAC (passes) then service=plumbing (stops). Now add a Router module — this sends data down different paths based on conditions. Understand why you'd use a filter vs. a router.
The HTTP Module — Your Most Important Tool
  • Why the HTTP module matters: Make has 1,000+ pre-built integrations, but the HTTP module lets you connect to ANY API that exists — including ones Make doesn't have a native module for. This is what you'll use for Claude. Add an HTTP module to a scenario. Look at all the fields: URL, Method, Headers, Body. These map directly to what you learned in Week 1.
  • Build Scenario #2 — HTTP to a real API: Use the HTTP module to call https://jsonplaceholder.typicode.com/posts/1 (GET request, no auth needed). Map the "title" field from the response into a text aggregator or set variable. You're now calling an external API from inside Make.
  • Reading scenario execution history: Run your scenario, then click the numbered bubble that appears on each module. This shows you exactly what data went in and what came out. This is your primary debugging tool — you'll live here when things break. Spend 10 minutes just clicking through execution history on a successful run.
  • Write in Journal: Draw the data flow of your webhook scenario on paper — what enters each module, what leaves it, what gets filtered or transformed. Then explain Make.com to someone (or yourself out loud) as if they've never heard of it.

WK 03 Airtable as a Real Database — Not a Spreadsheet ✅ 0/7 5–6 hrs ▼
Goal: Most people use Airtable like a prettier spreadsheet. You're going to use it like a database — with proper field types, relational links, views, and API access. This is the data layer for everything you build.
Database Thinking vs. Spreadsheet Thinking
  • Why this distinction matters: In a spreadsheet you repeat data (customer name in every row). In a database you store it once and reference it. Airtable lets you do both, but treating it as a database makes your automations cleaner, faster, and easier to debug. Read Airtable's "Linked records" guide (15 min).
  • Build your HVAC data structure: Create a new base called "Buster's H&A — Operations". Create 2 tables: "Customers" (Name, Phone, Email, Address, Created Date) and "Jobs" (Job ID, Customer link, Service Type, Urgency, Status, Notes, Date Created, Follow-Up Sent). The Jobs table links to Customers — one customer can have many jobs. This is a one-to-many relationship.
  • Field types matter: Use Single line text, Phone, Email, Date, Single select (for Status and Urgency — not free text), Linked record (for Customer), Checkbox (for Follow-Up Sent). Each field type validates and formats data differently. Wrong field type = broken automations later. Set up all fields with the correct types.
  • Views: Create 3 views in the Jobs table: Grid (all jobs), Gallery filtered to Status = "New" (incoming queue), and a Kanban grouped by Status. Views don't change the data — they change how you see it. Your clients will use these to manage their business.
Airtable's API — How Make Talks to It
  • Get your Airtable API key and Base ID: Go to airtable.com/create/tokens → create a personal access token with scopes: data.records:read, data.records:write, schema.bases:read. Also find your Base ID in the URL when viewing your base (starts with "app"). Write both down — you'll need them in Week 5.
  • Test the Airtable API directly in Hoppscotch: Make a GET request to https://api.airtable.com/v0/YOUR_BASE_ID/Jobs — add a Header: Authorization: Bearer YOUR_TOKEN. You'll get back JSON of your records. This is exactly what Make does when it reads from Airtable. You're seeing behind the curtain.
  • Now make a POST request to create a new record in Jobs via Hoppscotch. Body: {"fields": {"Service Type": "AC Repair", "Urgency": "Emergency", "Status": "New"}}. Check your Airtable base — the record appeared. You just wrote data to a database with an API call. This is what your automation will do thousands of times.

WK 04 The Claude API — How AI Fits Into an Automation ✅ 0/8 6–7 hrs ▼
Goal: Understand exactly how Claude works as an API — not as a chat interface. Learn how system prompts control behavior, how to structure requests for reliable outputs, and why this is the intelligence layer that makes your automations smarter than anyone else's.
How the Claude API Works
  • The Messages API structure: Every Claude API call has 3 parts — a model (which Claude version), a system prompt (permanent instructions that shape every response), and a messages array (the conversation). Read the Anthropic Messages API overview at docs.anthropic.com — focus on the "Request body" section. Don't skip this — it's 20 minutes that saves hours of confusion later.
  • Get your API key: Go to console.anthropic.com → API Keys → Create Key. Copy it somewhere safe. This key costs money per call (fractions of a cent each) — you'll use haiku-4-5 model which is the fastest and cheapest for automation tasks.
  • Make your first Claude API call in Hoppscotch: POST to https://api.anthropic.com/v1/messages. Headers: x-api-key: YOUR_KEY, anthropic-version: 2023-06-01, Content-Type: application/json. Body: {"model": "claude-haiku-4-5-20251001", "max_tokens": 100, "messages": [{"role": "user", "content": "Classify this service request as Emergency, Routine, or Quote: My AC stopped working and it's 95 degrees outside"}]}. Read the response — find the "text" field inside "content". That's your output.
  • System prompts — the most important concept: A system prompt tells Claude who it is and how to behave in every message. Without one, Claude is a general assistant. With one, it's a specialized tool that outputs exactly what your automation needs. Add a "system" field to your request body: "You are a job triage assistant for an HVAC company. Classify service requests as exactly one of: Emergency, Routine, or Quote. Respond with only the classification word, nothing else." Now your Claude call returns a single clean word your automation can act on.
Prompt Engineering for Automation
  • Why automation prompts are different from chat prompts: In chat you want rich, conversational responses. In automation you want predictable, structured outputs your code can parse. Key rules: (1) Tell Claude the exact format to respond in. (2) Give examples of correct output. (3) Tell it what to do when uncertain. (4) Never let it add explanations you didn't ask for.
  • Test your urgency classifier with 10 inputs: Try varied service requests — some clearly emergency, some routine, some ambiguous. Does Claude always return exactly one clean word? If not, tighten the system prompt. Add: "If you are uncertain, classify as Routine. Never add explanations, punctuation, or extra words." Test until 10/10 return clean single-word responses.
  • Build a second prompt — SMS personalizer: System prompt: "You write short, friendly SMS messages for an HVAC company. Write a 1-sentence follow-up message using the customer's first name and the service type they received. Maximum 160 characters. No emojis." User message: "Customer: Sarah. Service: furnace tune-up." Test it. You now have a reusable AI module for personalizing any message.
  • Write in Journal: What's the difference between a system prompt and a user message? Why does max_tokens matter in automation? What happens if Claude exceeds 160 characters on an SMS — and how does your system prompt prevent it?

WK 05 Twilio SMS — How Programmable Messaging Works ✅ 0/7 5–6 hrs ▼
Goal: Understand Twilio's architecture — phone numbers, messaging services, webhooks for inbound messages — and send your first programmatic SMS. This is the piece that makes your system feel real to clients.
How Twilio Works Under the Hood
  • Twilio's architecture: Twilio is a telecom API company. They own phone number infrastructure globally. You rent a number from them ($1/mo), then use their API to send/receive SMS, calls, and WhatsApp from that number. Your automation talks to Twilio's API → Twilio delivers to the carrier → carrier delivers to the customer's phone. Create a free Twilio trial account at twilio.com.
  • Key Twilio concepts: Account SID (your account identifier), Auth Token (your API password — never expose this publicly), From number (the Twilio number you send from), To number (the recipient). Get your Account SID and Auth Token from the Twilio Console dashboard. Buy a trial phone number (free on trial account).
  • Send your first SMS via Hoppscotch: POST to https://api.twilio.com/2010-04-01/Accounts/YOUR_ACCOUNT_SID/Messages.json. Auth: Basic auth with Account SID as username, Auth Token as password. Body (form-encoded, not JSON — Twilio uses form data): To=YOUR_MOBILE_NUMBER&From=YOUR_TWILIO_NUMBER&Body=Hello from Buster's H&A — your job has been received. You should receive a real SMS on your phone within seconds.
  • Inbound messages and webhooks: When a customer texts your Twilio number back, Twilio sends a webhook (a POST request with the message data) to a URL you configure. This is how you'd build a two-way conversation. In Twilio Console → Phone Numbers → your number → Messaging → set the webhook URL to your Make.com webhook URL from Week 2. Now any text to that number fires your Make scenario.
Twilio in Practice
  • Character limits and encoding: Standard SMS is 160 characters. Messages over 160 are split into segments and each costs separately. Unicode characters (emojis, curly quotes) reduce the limit to 70 characters per segment. This is why your Claude SMS prompt specifies "no emojis" and "maximum 160 characters" — you're protecting your client's costs.
  • Opt-out compliance: TCPA law requires that any marketing SMS include opt-out language ("Reply STOP to unsubscribe") and that you honor STOP replies. Twilio handles STOP responses automatically — they blacklist the number. Your follow-up messages should include "Reply STOP to opt out" in the first message of any sequence. This is a best practice you'll teach clients.
  • Write in Journal: What's the difference between a Twilio webhook (inbound) and a Twilio API call (outbound)? Why does SMS character limit matter for your Claude prompt design? What would happen to your client's Twilio bill if someone used emojis in every message?

WK 06 Connect Everything — Build the Full Intake System ✅ 0/7 6–7 hrs ▼

Build the Intake Scenario

  • Add the Claude HTTP module: POST to Anthropic API. System prompt: "You are a job triage assistant for an HVAC company.   • Add the Airtable HTTP module: POST to your Airtable Jobs table. Map fields: Service Type from Tally dropdown, Urgency from Claude's response text, Status = "New" (hardcoded), Customer name and phone from Tally. Create the record.
  • Add the Twilio HTTP module: POST to Twilio Messages API. To = your phone number (hardcoded for now). From = your Twilio number. Body = "🔧 New job — [Customer Name] at [Address]. Issue: [Description]. AI Urgency: [Claude output]. Call to confirm." Map all values from earlier modules.

WK 07 Build the Follow-Up Sequence + Review Request ✅ 0/6 5–6 hrs ▼
Goal: Build a second scenario that runs on a schedule — reading Airtable for jobs that need follow-up and sending timed SMS sequences. This introduces scheduled triggers, Airtable reads, and multi-step logic.
The Follow-Up Scenario
  • How scheduled scenarios work: This scenario doesn't wait for a form submission. It runs every hour (or day), reads Airtable for records that match certain conditions, and acts on them. Trigger: Make "Schedule" module set to run every 1 hour. This is a polling pattern — common in automation.
  • Add Airtable "Search Records" module: Search your Jobs table for records where Status = "Complete" AND Follow-Up Sent = false (unchecked). These are jobs that are done but haven't been followed up yet. Map the results through an Iterator module (this processes multiple records one at a time).
  • Send the follow-up SMS: For each record, send a Twilio SMS to the customer's phone: "Hi [First Name], thanks for choosing Buster's Heating & Air! How did your [Service Type] service go? Reply anytime if you have questions." Then update the Airtable record: set Follow-Up Sent = true so it won't send again.
Review Request + Timing
  • Add a date-based filter for the review request: Add a second path after the follow-up. Check if the job's Date Created was 3 days ago (use Make's date functions: formatDate and now). If yes, send a second SMS: "Hi [First Name], if Buster's took good care of you, a quick Google review means the world to a small business: [Google Review URL]. Thanks!"
  • Test by manually setting a job's Status to "Complete", Follow-Up Sent to false, and Date Created to 3 days ago. Run the scenario manually (don't wait for the scheduler). Confirm both SMS messages fire correctly and the Airtable record updates.
  • Write in Journal: Why does the "Update Record" step (setting Follow-Up Sent = true) happen AFTER the SMS sends and not before? What would go wrong if you reversed the order? This is a sequencing principle that matters in every automation you'll ever build.

WK 08 Debugging — How to Find and Fix Anything That Breaks ✅ 0/7 5–6 hrs ▼
Goal: Deliberately break your system in 5 different ways and fix each one. This is the most valuable week in the program. Every client conversation where you say "I know exactly why that broke and how to fix it" comes from this practice.
Break It On Purpose
  • Break #1 — Bad API key: Replace your Anthropic API key in the HTTP module with a wrong value. Run the scenario. Read the error in execution history — what does a 401 Unauthorized error look like? Fix it. Now you'll recognize auth errors instantly in real deployments.
  • Break #2 — Empty required field: Submit the Tally form without filling in the phone number. What happens in Make? Does the scenario fail, skip, or create a record with an empty field? Add a filter after the Tally trigger: only proceed if Phone Number is not empty. Test it. This is input validation — you'll add it to every client system.
  • Break #3 — Airtable field name mismatch: In your Airtable HTTP module, change "Service Type" to "service type" (lowercase). Run it. Read the error. Fix it. Field names in Airtable's API are case-sensitive — this is one of the most common real-world bugs you'll encounter.
  • Break #4 — Claude returns unexpected output: Remove the strict formatting instruction from your system prompt ("No other output"). Submit a form with an ambiguous description. Does Claude add explanation text? How does that break your Airtable urgency field? Restore the strict prompt. This demonstrates why prompt precision matters.
Debug Methodology
  • Break #5 — Twilio wrong number format: Change the "To" phone number to a format without the country code (use 8015551234 instead of +18015551234). What error does Twilio return? Fix it. Phone number formatting is a consistent source of client bugs — you now know to always verify +1 country code prefix.
  • Add error handlers: In Make, right-click on a module → Add error handler → Ignore (for non-critical steps like SMS — you don't want the whole scenario to fail if one SMS doesn't send). Add a separate error notification: if the Airtable write fails, send yourself an email with the error details. This is how professional systems handle failure gracefully.
  • Write in Journal: Create a personal "Break/Fix log" — for each of the 5 breaks, write: what the error message said, what the actual cause was, and what you changed to fix it. This document is gold when you're supporting client systems at 10pm.

Objection Handling
"We already have software that does this."
→ "Does it automatically follow up on every missed call, triage urgency with AI, and send a Google review request 3 days after every completed job?" Most shop management systems don't. That's the gap I fill. I'm not replacing your software — I'm adding the intelligence layer on top of it.
"We're too small for something like this."
→ "That's actually why this works so well for you." A 2-truck shop missing 3 emergency calls a week loses $15–20K in annual revenue at $450/call. This pays for itself in the first month if it catches 2 jobs you'd have lost.
"What if it breaks?"
→ "That's what the retainer covers." I monitor it, fix errors same day, and you get a monthly check-in. If something breaks and I don't catch it in 24 hours, your next month is free. That's the service agreement.
"We tried automation before and it didn't work."
→ "What went wrong?" (Let them answer.) Usually it's that someone set it up once and never maintained it, or it was too generic. This is configured specifically for how your shop operates, and I manage it ongoing. That's different from a one-time tool setup.
"I need to think about it / talk to my partner."
→ "That makes sense. What's the main thing you'd want to talk through?" Get the real objection out. Then: "Want me to set up a 15-minute call with both of you so you can see it live?" Give them a deadline: "I can hold your spot in the onboarding queue until [specific date]."
